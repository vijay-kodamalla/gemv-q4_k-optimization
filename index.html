6 CUDA kernel variants on an A100, obsessive Nsight Compute profiling, and the counterintuitive reason why the "naive" upstream kernel beats everything I tried.

Every time a quantized LLM generates a token, a matrix-vector multiply runs against the full weight matrix. In llama.cpp's Q4_K format, this means dequantizing 4-bit weights and computing dot products simultaneously—at an arithmetic intensity of just 2 ops/byte. On an A100 with ~1900 GB/s of bandwidth, that puts us firmly in the memory-bound regime.

I wanted to understand this kernel at the deepest level. Not by reading code—by writing it from scratch, profiling every variant with Nsight Compute, and figuring out what actually makes it fast or slow on real hardware.

I explored 6 kernel variants. I got within 11% of the production upstream kernel (22.27 μs vs 20.03 μs). And I learned that almost everything I thought I knew about GPU optimization inverts for memory-bound workloads.

The Q4_K Constraint

Here's the thing about Q4_K: each block stores 256 weights in 144 bytes. The dequantization math (vec_dot_q4_K_q8_1) takes an index iqs from {0, 2, 4, …, 30}—exactly 16 possible values. That means the math kernel needs exactly 16 threads. Not 32. Sixteen.

So the upstream llama.cpp kernel just… leaves threads 16–31 idle. Fifty percent of the warp doing nothing. My first reaction was: surely we can do better than that?

Spoiler: it's more complicated than I thought.

Variant 1: Weight Tiling

The textbook optimization—tile weights into shared memory, reduce global memory traffic. I loaded 64 Q4_K blocks (9 KB) into shared memory per CTA.

It was slower. Here's what Nsight Compute told me:

→ 32.67 μs, 34,461 cycles, 48 registers/thread
→ Compute throughput: 43.5%  |  Memory throughput: 38.36%
Ncu complains about a latency issue—both are below 60%
→ ALU pipeline utilization: 33.4%
→ Scheduler: 1 instruction per 2 cycles
Every scheduler is capable of issuing 1 instruction per cycle, but here it's 1 per 2. The warps are stalling somehow.
→ Stall: each warp spends 4.2 cycles waiting on L1TEX scoreboard
→ Theoretical occupancy: 45.3%  |  Achieved: 31%
→ L1TEX global load problem: only 15.4 of 32 bytes per sector utilized
Ncu estimates a 20% speedup if this access pattern is fixed
→ Uncoalesced accesses: 2,621,440 excessive sectors (62% of 4,235,264 total)
→ Instructions executed: 6,434,816
→ Grid: (4096,1,1) × (32,1,1)

The problem was simple in hindsight: GEMV has no data reuse. Each weight is used exactly once. In GEMM, you tile because the same weight gets reused 128 times across the output tile. In GEMV? You load the weight, multiply it, and never touch it again. So I was paying all the shared memory overhead—the occupancy hit, the L1 cache displacement, the bank conflicts—and getting nothing back.

And look at that occupancy: theoretical 45.3%, achieved only 31%. The shared memory allocation was the limiter. On A100, shared memory and L1 share the same physical SRAM—so more shared memory literally means less L1 cache. Hit rate dropped from around 80% to 30%.

Lesson: don't tile without reuse. Tiling is not universally good. It's specifically good when you amortize the overhead over many reuses. GEMM has 128× reuse. GEMV has 1×. The math doesn't work.

Variant 2: Activation Tiling

Okay, weights don't have reuse. But the activation vector gets reused across every output row, right? So I staged it in shared memory.

This was catastrophic.

→ 80.58 μs, 85,800 cycles, 40 registers/thread
→ Compute throughput: 19.83%  |  Memory throughput: 19.20%
Both below 20%! Ncu is screaming about latency issues.
→ 1.7-way bank conflicts, as expected
→ Scheduler: 1 instruction per 4.6 cycles
4.6 cycles per instruction! The warps are stalling catastrophically.
→ Stall: each warp spends 5.6 cycles waiting on L1TEX scoreboard
→ Theoretical occupancy: 12.5%  |  Achieved: 11.82%
→ Active warps per scheduler: 2  |  Eligible: 0.25
0.25 eligible warps! The scheduler has almost nothing to issue.
→ Instructions executed: 7,331,840
→ Grid: (4096,1,1) × (32,1,1)

Look at those eligible warps: 0.25 per scheduler. The scheduler literally has nothing to issue. Occupancy at 11.82% means almost no warps are resident to hide latency.

But here's the real kicker: the activation vector for Mistral 7B is about 128 KB. The A100 has a 40 MB L2 cache. The activations were already cached. I was staging L2-cached data into shared memory, adding 1.7-way bank conflicts and 32 __syncthreads() barriers (32 × 40 cycles ≈ 1.28 μs of pure overhead) for absolutely zero benefit.

Lesson: don't stage what's already cached. Before you reach for shared memory, check if the data already lives in L2. If it does, you're adding overhead for nothing.

Variant 3: The 2pack Strategy

After two failures, I took a step back. The Q4_K math needs 16 threads. I have 32 threads in a warp. Instead of trying to rewrite the math or tile data into shared memory, what if I just give the upper 16 threads a second block to work on?

Threads 0–15 process block K. Threads 16–31 process block K+1. Same math kernel, different data. Loop stride goes from 1 to 2—half the iterations.

2pack core idea
const int logical_lane = lane_id & 0xF;   // 0-15 for everyone
const int block_offset = lane_id >> 4;     // 0 for lower half, 1 for upper
const int my_kbx = kbx + block_offset;     // Lower: block K, Upper: block K+1

for (int kbx = start; kbx < end; kbx += 2) {
    sum += vec_dot(W, Y, my_kbx, iqs);
}

And then the reduction needs an extra step—merge the two halves before the standard warp reduction:

2pack reduction
// Step 1: Merge upper and lower halves
sum += __shfl_xor_sync(0xffffffff, sum, 16);

// Step 2: Standard reduction within lower 16
sum += __shfl_down_sync(mask16, sum, 8);
sum += __shfl_down_sync(mask16, sum, 4);
sum += __shfl_down_sync(mask16, sum, 2);
sum += __shfl_down_sync(mask16, sum, 1);
// Thread 0 has the complete warp sum

Now look at these numbers:

→ 22.27 μs, 23,239 cycles, 48 registers/thread
→ Compute throughput: 62.04%  |  Memory throughput: 47.50%
First time compute is above 60%! Actually utilizing the hardware.
→ ALU pipeline utilization: 48.2%
→ Theoretical occupancy: 62.5%  |  Achieved: 52.61%
→ Active warps per scheduler: 8.52  |  Eligible: 2.68
10× more eligible warps than activation tiling. The scheduler is happy.
→ Instructions executed: 7,331,840
Same instruction count as Y-tiling, but look at the throughput difference.
→ Uncoalesced: 2,949,120 excessive sectors (62% of 4,792,320 total)
→ Grid: (4096,1,1) × (32,4,1)

This beat every tiling and async variant by 25–35%. Compute throughput crossed 60% for the first time. Occupancy jumped to 52.61%—with zero shared memory overhead. All 32 threads doing useful work.

But: it's still 11% slower than upstream.

The Other Attempts

I tried two more variants that combined weight tiling with 4 warps per CTA—one with a single CTA layout, one splitting across 2 CTAs. Both landed around 29.8–29.9 μs. Interestingly, they had the highest occupancy and compute throughput numbers (53.72% and 62.86% respectively), but it didn't matter—the shared memory overhead for weight tiles still dragged them below 2pack.

I also noticed L2 slice imbalance in the Nsight Compute reports: some L2 slices were 23.5% more active than the average. Uneven memory traffic across the L2 partitions—another subtle cost of the tiling approach.

Stare at the 2pack and upstream rows. 2pack has better occupancy (52.61% vs 34.56%), better compute throughput (62.04% vs 53.58%), better memory throughput (47.50% vs 36.46%), and uses fewer registers (48 vs 66).

And it's still slower.

The answer is in the instructions column: 7,331,840 vs 4,780,032. That's 53% more instructions. And at an arithmetic intensity of 2 ops/byte, every single instruction matters.

Why the "Naive" Kernel Wins

The upstream kernel is dead simple. One block per warp iteration. Only 16 of 32 threads active. No indexing tricks. No half-warp merging. Straight for loop.

And it produces 4,780,032 instructions—the lowest of any variant.

The instruction gap between 2pack and upstream is 2,551,808. At roughly 1 cycle per instruction on A100, that's about 2.55 μs. The measured time gap is 2.24 μs. That's an 88% match—the instruction overhead almost entirely explains the performance difference.

Where do the extra instructions come from in 2pack?

→ Indexing per iteration: lane_id & 0xF, lane_id >> 4, kbx + block_offset, boundary check
→ Reduction: extra __shfl_xor_sync to merge halves + more complex mask management
→ Non-fused FP32: 2pack has 327,680 non-fused vs upstream's 196,608
More non-fused instructions means the compiler couldn't combine as many ops—the complex indexing is tying its hands.
→ Compiler: more live values → less freedom to schedule → worse code generation

And notice something counterintuitive: upstream uses 66 registers per thread—the most of any variant. But it doesn't matter because the code is so simple that the compiler can use those registers effectively. My 2pack uses only 48 registers but has more register pressure from the complex indexing, which is a different problem entirely.

At arithmetic intensity = 2, instruction count is everything. Occupancy, warp utilization, compute throughput—none of it matters enough to overcome a 53% instruction overhead. The kernel is so memory-starved that the bottleneck is how many instructions the GPU has to chew through between memory transactions.

The Persistent Problem: Uncoalesced Loads

One thing that showed up in every single variant: uncoalesced global memory access when loading activations in the math kernel. Look at the excessive sector percentages across the board—62%, 62%, 62%, 62%, 60%. Nearly identical regardless of what optimization I tried.

In the weight tiling variant, Nsight Compute explicitly flagged this: on average, only 15.4 of the 32 bytes transmitted per sector are actually utilized by each thread, with an estimated 20% speedup if fixed.

The root cause is structural. In Q4_K format, when threads in a warp dequantize different weight blocks, the corresponding activation elements they need are strided in memory. Each 32-byte sector load wastes about half its bytes. That's inherent to how the Q4_K block layout interacts with warp-level parallelism—and it affects the upstream kernel (60% excessive) just as much as mine (62% excessive).

This is a real optimization opportunity: cooperative warp loading of activations into shared memory with a coalesced access pattern could recover that 20%. But it would add shared memory usage and synchronization, which as we've seen is dangerous in memory-bound territory. That's a tradeoff I'm still thinking about.

The Tradeoff Framework

After six variants and hundreds of profiling runs, I started to see a pattern. Every GPU optimization trades something for something else. The trick is knowing which resource is abundant and which is scarce.

Trade what is abundant for what is scarce. In GEMV (AI = 2): parallelism is abundant, instruction budget is scarce—keep it simple. In GEMM (AI = 128): instruction budget is abundant, data bandwidth is scarce—tile aggressively.

Trade Helps When Hurts When
More occupancy ↔ More smem Compute-bound (hide latency) Memory-bound (smem steals L1)
More parallelism ↔ More instructions High AI (compute hides overhead) Low AI (every instruction costs)
More data reuse ↔ More smem overhead High reuse (GEMM: 128×) No reuse (GEMV: 1×)
More async ↔ More complexity High compute/memory ratio Low compute/memory ratio

And here's the thing: the same 2pack strategy that costs me 11% on GEMV would win on GEMM. The 2.55M extra instructions are 0.002% of GEMM's 137 billion FLOPs—completely negligible. But the occupancy boost from 34% to 52% would give 15–20% more Tensor Core utilization. The technique isn't bad. I was applying it to the wrong workload.

Knowing When to Stop

After reaching competitive GEMV performance, I did something I probably should have done at the start: an impact analysis.

Kernel % of Inference Optimization Headroom End-to-End Impact
GEMM ~90% 2× (40% → 80% of peak) ~1.8× speedup
Attention ~5% Moderate ~2–5%
GEMV ~3% 30% ~0.9%

GEMV is 3% of inference time. Even if I somehow matched upstream perfectly—a 10% GEMV improvement—that's a 0.3% end-to-end gain. Meanwhile, GEMM is 90% of the runtime. The same engineering skills applied to GEMM have 200× more leverage.

I stopped GEMV optimization. Not because I ran out of ideas—the uncoalesced activation load pattern is still sitting there with a potential 20% improvement. I stopped because the Pareto principle told me to pivot.

The mature engineering decision isn't always "optimize harder." Sometimes it's recognizing where your skills have the most impact. The techniques that failed for GEMV—tiling, async pipelining, occupancy optimization—are exactly the techniques that dominate GEMM. Understanding when each technique applies is the real skill.

What I Took Away

Arithmetic intensity determines your entire optimization strategy. At AI = 2, minimize instructions. At AI = 128, maximize compute utilization. The same technique can help at one and hurt at the other.

Profile with Nsight Compute, not with intuition. I was convinced tiling would help. The numbers said otherwise. The scheduler statistics, the eligible warp counts, the L1 hit rates—these told the real story. When I saw "1 instruction per 4.6 cycles" on the activation tiling variant, that was the kind of signal you can only get from actual profiling.

Don't tile without reuse. Don't stage what's cached. Don't async without compute to hide behind. Every "best practice" has a regime where it becomes an anti-pattern. Knowing the boundary conditions is what separates reading GPU optimization blogs from doing GPU optimization.

The compiler is your best optimizer when you give it simple code. Upstream uses the most registers (66) of any variant and still produces the fewest instructions (4.78M). Simple code lets the compiler do its job.

Know when to stop and where to apply your skills next. GPU optimization is a tradeoff game—profile, identify the bottleneck, trade abundant resources for scarce ones, verify, iterate. And always ask: am I optimizing the right kernel?
